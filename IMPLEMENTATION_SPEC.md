## 一、核心理念（必须理解）

### 1.1 产品定位

Book Lifeline 不是 AI 找书，也不是读书笔记管理，而是 **AI 找回用户曾经在书里留下的自己**。

真正被保存的不是卡片，而是 **"当前的我"与"过去在书中留下痕迹的我"之间被重新确认的关系**。

### 1.2 内容分类体系

| 分类 | 定义 | 来源 | 在来源页中的位置 |
|------|------|------|-----------------|
| **读书摘要** | 来自阅读平台的内容：划线、想法、摘要 | 微信读书等 | 第一部分，按书分组 |
| **日记** | 真实的生活和观察记录 | Notion / 日记应用 | 第二部分，按日期分组 |
| **书脉输入** | 用户在 Book Lifeline 中记录的原始感受 | Book Lifeline | 第三部分，按时间倒序 |

**没有新旧节点之分**。所有输入内容一旦保存就是旧节点，都可以被未来的新输入关联。

### 1.3 "感受" vs "想法"

用户在 Book Lifeline 中输入的内容统一称为 **"感受"**。

- 「想法」是加工过的、偏理性的成品 → 不使用
- 「感受」是原初的、第一人称的体验单位 → 统一使用

外部平台原有术语保留（微信读书的"想法"、Notion 的"笔记"等）——在来源页展示时标注原始来源。

### 1.4 关系类型

| 英文 | 中文 | 含义 |
|------|------|------|
| echo | 回响 | 此刻感受与旧痕迹在深层结构上呼应 |
| contrast | 对照 | 旧痕迹与当前感受形成对比 |
| source | 溯源 | 旧痕迹是理解当前感受的源头 |
| example | 例证 | 旧痕迹作为当前感受的实例 |
| question | 追问 | 旧痕迹引出新问题 |
| correction | 修正 | 旧痕迹修正或补充当前理解 |

### 1.5 关系方向

- **个人经验型**：我的感受 ↔ 我过去的痕迹（纵向，时间轴上的自我对话）
- **引用传承型**：书A的概念 ↔ 书B的概念（横向，知识网络中的谱系发现）

Demo 阶段主要实现个人经验型关系。引用传承型作为 v2 方向。

## 二、交互设计

### 2.1 页面清单（10个页面）

| 页面 | 文件名 | 核心功能 |
|------|--------|---------|
| 感受页 | `index.html` | 用户输入感受的起点 |
| 节点确认页 | `节点确认页.html` | AI 对感受的结构化理解确认 |
| AI 分析中 | `AI 分析中.html` | 旧痕迹检索动画（3.5秒自动跳转） |
| 旧回声推荐页 | `旧回声推荐页.html` | 展示找到的 old traces + 关系解释 |
| 新节点确认页 | `新节点确认页.html` | 无旧痕迹时的独立节点确认 |
| 书脉回看页 | `书脉回看页.html` | 已确认关系的可视化回看 |
| 书脉总览页 | `书脉总览页.html` | 全部书脉的概览 + 模式总结 |
| 书籍详情页 | `书籍详情页.html` | 单本书的连接详情 |
| 来源页 | `来源页.html` | 三部分痕迹出处 |

### 2.2 底部导航栏（所有页面统一）

```
记录 ｜ 书脉 ｜ 来源
```

- 书脉放在中心位置（移动端拇指热区）
- 当前所在 Tab 高亮（--blue），其余灰色
- Tab 图标保持现有设计（不需要重新设计）

### 2.3 核心流程（两种路径）

**路径 A：找到旧痕迹**
```
感受页 → 节点确认页 → AI 分析中（自动跳转）→ 旧回声推荐页 → 书脉回看页
```

**路径 B：未找到旧痕迹**
```
感受页 → 节点确认页 → AI 分析中（自动跳转）→ 新节点确认页 → 书脉总览页
```

**来源页**和**书籍详情页**通过 Tab 导航和点击卡片进入，不在主流程中。

### 2.4 旧回声推荐页 — 卡片信息层级（从上到下）

```
第一层：旧痕迹本身（最重要）
  "你曾经在《世界简史》中划线标记过："
  "那些发生在过去的互动仍深刻影响着今天..."

第二层：来源信息
  来自：《世界简史》· 威廉·麦克尼尔 微信读书划线 · 2024.12.01
  [溯源]

第三层：关系解释
  为什么与此刻相关
  "你刚刚写到'把分散的东西连接成一个系统'，
   而这条你半年前在《世界简史》中划线标记的句子..."

第四层：用户动作
  [确认这条关系]        ← 主按钮，全宽
  [放弃]    [改写理由]   ← 次按钮，并排
```

### 2.5 新节点确认页 — 文案结构

**变体 A：与书有关**
```
标题：你的新感受已保存为书脉输入
副标题：它将成为未来其他感受关联的起点

卡片：用户输入的内容 + 书名 + 标签

关于这次结果：
  这次没有找到与你过去记录明显相关的旧痕迹。
  但这很正常——这是一条全新的读书感受。
  它此刻是独立的，但未来某天，当你写下新的感受时，
  AI 可能会在那时找回这一条。

信任声明：
  你在 Book Lifeline 中输入的所有内容都安全地保存在你的个人空间中。

来源提示：
  这条书脉输入可以在来源页的'书脉输入'区域中查看。

按钮：[继续记录]    [进入书脉]
```

**变体 B：与生活有关**（类似，标题为"你的生活感受已保存为书脉输入"）

## 四、数据结构

### 4.1 ReadingNode（AI 解析后的结构化感受）

```typescript
interface ReadingNode {
  id: string;                    // 唯一标识
  rawText: string;               // 用户原始输入
  type: 'insight' | 'question' | 'observation' | 'reflection';
  tags: string[];                // AI 提取的标签
  bookTitle?: string;            // 关联书籍（可选）
  bookAuthor?: string;           // 作者（可选）
  createdAt: string;             // ISO 日期
}
```

### 4.2 OldTrace（旧痕迹）

```typescript
interface OldTrace {
  id: string;
  content: string;               // 痕迹内容（划线文本、想法内容等）
  source: 'wechat_reading' | 'notion' | 'booklifeline';
  sourceType: 'bookmark' | 'thought' | 'diary' | 'booklifeline_input';
  bookTitle?: string;
  bookAuthor?: string;
  createdAt: string;
}
```

### 4.3 EchoRelation（旧回声关系）

```typescript
interface EchoRelation {
  id: string;
  readingNodeId: string;         // 当前感受
  oldTraceId: string;            // 关联的旧痕迹
  relationType: 'echo' | 'contrast' | 'source' | 'example' | 'question' | 'correction';
  reason: string;                // AI 生成的关系解释
  confirmed: boolean;            // 用户是否确认
}
```

### 4.4 Lifeline（书脉）

```typescript
interface Lifeline {
  id: string;
  readingNode: ReadingNode;
  relations: EchoRelation[];
  createdAt: string;
}
```
