# Book Lifeline Demo UI 交互重设计指令

## 文档定位

本文件是一份**界面设计调整指令**，不是新的 PRD。

你已经为 Book Lifeline 设计过 6 个静态 HTML 页面。现在产品理念经过了校准，需要你根据新理念**调整现有页面的 UI 交互设计**。你不需要从零开始设计，而是在已有设计的基础上做精准修改。

**本文件中的所有示例内容均来自真实数据**——用户的微信读书划线、读书想法，以及 Notion 中的阅读笔记。

---

## 0 唯一的产品核心

理解这一句话，就理解了所有设计决策的依据：

> Book Lifeline 不是 AI 找书，也不是生成漂亮卡片，而是 AI 找回用户曾经在书里留下的自己。

**旧回声**的定义被严格限定为：**当前输入与用户旧痕迹之间被 AI 找出的可解释关系**。

旧痕迹特指用户过去主动留下过的内容——微信读书划线、读书想法、日记、书脉输入等。旧痕迹有三个来源：**读书摘要**（来自阅读平台）、**日记**（来自日记工具）、**书脉输入**（来自 Book Lifeline）。

用户确认的也不是一张卡片或一本书，而是一条**关系**——"此刻的我"与"过去在书中留下痕迹的我"之间的连接。

## 3 旧回声推荐页 — 重点修改

这是理念校准后变化最大的页面。

### 3.1 演示场景设定

为了让设计 Agent 理解卡片要展示什么内容，这里设定一个具体的演示场景：

**用户当前输入**（在输入页写下的一段读后感）：

> 刚读完《世界简史》，麦克尼尔说"把人类历史作为一个整体来进行一番概览"，这句话本身平淡，但实际指向的，可能是人类社会组织方式的一次此后难以逆转的结构性变化。那一千年里，人类不只是多了城市、文字、神庙，而是第一次把这些东西连接成一个系统。我想到我自己的工作——我们也不是在做一个单一功能，而是在试图把分散的东西连接成一个系统。

**AI 从用户的旧痕迹中找到了 3 条相关内容**——下面是卡片中应该展示的真实内容。

### 3.2 卡片信息层级 — 从上到下

### 3.3 按钮文案与排列

**现在**：`保留` | `删除` | `改写理由`

**改为**：

主按钮（较大，填充色 `--blue`）：
```
确认这条关系
```

次按钮（较小，背景 `--fill-secondary`，并排两个）：
```
放弃    |    改写理由
```

排列方式：主按钮独占一行（全宽），两个次按钮在下方并排。

### 3.4 分组标签

**现在**：`来自日记池 · 2 条` / `来自读书笔记池 · 1 条`

**改为**：`读书摘要 · 3 条`

**理由**：在演示场景中，3 张卡片均与书籍相关（卡片 1 来自《世界简史》划线，卡片 2 来自《深度思考》想法，卡片 3 来自《世界简史》书评），全部属于读书摘要。

**通用分组规则**：
- 如果召回的旧痕迹全部来自同一来源 → 统一标注"读书摘要 · N 条" 或 "日记 · N 条" 或 "书脉输入 · N 条"
- 如果召回的旧痕迹来自多个来源 → 分组标注，如"读书摘要 · 2 条 / 日记 · 1 条"

### 3.5 关系类型 Badge

保留现有的 6 种关系类型（echo / contrast / source / example / question / correction），但展示中文：

| 英文 | 中文 | Badge 颜色 |
|------|------|-----------|
| echo | 回响 | `--green` |
| contrast | 对照 | `--orange` |
| source | 溯源 | `--purple` |
| example | 例证 | `--blue` |
| question | 追问 | `--green` |
| correction | 修正 | `--orange` |

Badge 位置：放在第二层"来源信息"的右侧或下方，不要放在第三层关系解释的上方。它属于来源元信息，不属于关系解释。

### 3.6 底部确认区域与流程分支

**现在**：
```
确认要保留的旧回声关系
[确认选择，进入书脉]
没有找到相关的？跳过
```

**改为**：
```
确认要建立的关系
[确认关系，进入书脉]
```

**重要分支逻辑**：

旧回声推荐页只在 AI 找到旧痕迹时展示。如果 AI 没有找到足够明确的旧痕迹，**不展示此页面**，而是直接跳转到**新节点确认页**（见第 4 节）。

两个页面互斥：
- 找到旧痕迹 → 旧回声推荐页 → 用户确认关系 → 书脉回看页
- 没找到旧痕迹 → 新节点确认页 → 用户确认保存 → 书脉总览页

## 4 新节点确认页（新增）

### 4.1 页面定位

当 AI 分析完成后，没有找到与用户过去记录明显相关的旧痕迹时，展示此页面。

这不是"失败提示页"，而是**"独立节点确认页"**。核心信息：
- 用户的新感悟已被保存
- 没找到旧痕迹是正常的，不是系统故障
- 这条新内容有独立价值，并将成为未来其他感悟关联的起点
- 用户在 Book Lifeline 中输入的所有内容都属于自己

### 4.4 文案设计原则（两种变体通用）

- **确认感**：标题"你的新感悟已保存为..."让用户知道输入已被接收并归类
- **诚实**："这次没有找到与你过去记录明显相关的旧痕迹"——如实说明，不带歉意
- **安抚**："但这很正常——这是一条全新的..."——强调这是正常状态
- **展望**："它此刻是独立的，但它会成为未来其他感悟关联的起点"——让用户看到未来价值
- **信任**："你在 Book Lifeline 中输入的所有内容都安全地保存在你的个人空间中"——建立长期信任

### 4.5 按钮行为

- **进入书脉**：导航到书脉总览页，用户可以在那里看到这条新内容被单独展示
- **继续记录**：导航回输入页，用户可以写下新的感悟

### 4.6 与旧回声推荐页的关系

旧回声推荐页和新节点确认页互斥，由 AI 分析结果决定展示哪一个：

```
ReadingNode 确认页
        ↓
   AI 分析旧痕迹
        ↓
   ┌────┴────┐
 找到      没找到
   ↓          ↓
旧回声    新节点确认
推荐页      页
   ↓          ↓
书脉      书脉总览
回看页      页
```

**新节点的归类逻辑**：
- 与书有关（有 bookTitle）→ 保存到书脉输入，同时出现在来源页对应书籍的读书摘要下
- 与生活有关（无 bookTitle）→ 保存到书脉输入，出现在来源页"书脉输入"区域

所有输入一旦保存就是旧节点，都可以被未来的新输入关联。

## 5 来源页（新增）

### 5.1 页面定位

来源页是用户所有痕迹的出处空间，包含三个独立的部分：

1. **读书摘要**：来自微信读书等阅读平台的划线、想法、摘要
2. **日记**：来自 Notion、日记应用等工具的真实生活和观察记录
3. **书脉输入**：用户在 Book Lifeline 中输入的原始感受和想法

Book LifeLine 的核心是连接**书**与**生活**。这三个部分分别代表了"书"（读书摘要）、"生活"（日记）、以及"书与生活的交汇点"（书脉输入）。

**没有新旧节点之分**。所有输入内容一旦保存就成为旧节点，都可以被未来的新输入关联。书脉输入中与书有关的内容可以同时出现在读书摘要的对应书籍下。

这个页面建立的认知是：**旧回声不是凭空生成的，它来自你过去真实留下的东西——无论来自书、来自生活、还是来自你在书脉中写下的感悟。**

### 5.2 统计概览

```
约 10 本书籍    300+ 条读书摘要    15+ 条日记    3 条书脉输入
```
